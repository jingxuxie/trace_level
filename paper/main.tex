\begin{abstract}
Tool-using language-model agents are often protected one action at a time: a
tool call is checked for a valid recipient, a visible-content filter scans the
message body, or a policy prompt asks the model to ``be careful.'' These
mechanisms are useful, but many agentic security policies are trace properties.
They depend on where information came from, how it was transformed, which actor
received it, and whether an approval or aggregation rule still applies. We study
this \emph{local-permission/global-policy gap}: every individual tool call in a
workflow can be locally authorized and plausible, while the composed execution
violates a policy over provenance, audience, tenant boundaries, memory, approval
scope, or aggregation. We introduce TraceBreak, a synthetic benchmark of paired
safe and risky enterprise workflows, and TraceGuard, a deterministic provenance
monitor that propagates source tags through reads, summaries, memory, and write
sinks. On 120 deterministic paired tasks, local guards and visible-content DLP
permit all risky traces, while TraceGuard blocks all risky writes with no
safe-control false blocks. On a 24-task API subset with \texttt{gpt-4.1-mini},
local guards and DLP again produce local-pass violations on every risky task;
policy prompting reduces but does not eliminate violations; TraceGuard blocks all
risky sink attempts while preserving safe-control utility. Additional replay
audits show that model-written provenance references are not an adequate trust
boundary: deleting or corrupting sink references breaks ordinary TraceGuard,
whereas runtime-inferred provenance restores safety on the replay. The result is
a simple lesson for agent security: the execution trace, not only the next tool
call or the natural-language prompt, should be a first-class enforcement object.
\end{abstract}

\section{Introduction}

Language-model agents are moving from answering questions to operating
workflows. A single task may require searching enterprise documents, reading
records, summarizing intermediate results, saving a handoff note to shared
memory, consulting an approval, and finally sending an email or updating a ticket.
This composition is precisely what makes agents useful, and precisely what makes
them hard to secure. The security question is no longer just ``is this tool call
well formed?'' It is often ``is the data that arrived here allowed to flow to
this sink, for this recipient, under this approval, after these transformations?''

Most lightweight agent safeguards answer the first question. A local tool guard
can check that a document ID exists, a recipient is known, or a source reference
is syntactically valid. A visible-content DLP rule can scan the current email
body for a secret-looking string. A prompt can instruct the model not to violate
policy. None of these mechanisms necessarily knows that the innocuous phrase
``short renewal update'' was derived from a confidential AtlasCo pricing memo
read four steps earlier, or that an approval was scoped to a different recipient.
The dangerous fact is distributed across the source object, the transformation
history, the recipient record, the approval, and the final sink.

This paper studies the resulting \emph{local-permission/global-policy gap}. A
trace has a local-pass violation when every executed tool call passes local
authorization, yet the full trace violates a global policy. The gap is not
limited to explicit prompt injection. It can arise in ordinary-looking workflows:
a confidential document is summarized before being emailed externally; a support
record for one tenant is used to update another tenant's ticket; a detailed list
of records is exported where only an aggregate is allowed; or a memory entry
launders the sensitivity of its source. The failure is also naturally
multi-agent. A researcher may read the source, a planner may summarize or store
it, and an executor may perform the final write. Each actor sees only a slice of
the policy state.

The security community has seen analogous problems before. Complete mediation
requires checking access whenever an object is used \citep{saltzer1975protection};
information-flow control tracks whether data may move between labeled contexts
\citep{denning1976lattice}; runtime monitors enforce trace properties
\citep{schneider2000enforceable}; and confused-deputy failures occur when
authority is silently separated from the object or request it should constrain
\citep{hardy1988confused}. Modern LLM agents recreate these classic concerns in
a new substrate: natural-language transformations, tool observations, memory, and
delegation can erase the connection between data and authority unless the runtime
preserves it.

We make the problem concrete with TraceBreak, a small synthetic benchmark of
paired safe and risky enterprise workflows. Each risky task has a matched safe
control that changes one policy-relevant field while keeping the surface task
nearly identical. This lets us measure both security and utility: a useful guard
should block the risky sink but allow the matched safe sink. We also implement
TraceGuard, a simple provenance monitor that attaches tags to tool outputs,
conservatively merges tags through transformations, and checks the merged tag
only when data reaches a sink.

Our contributions are:
\begin{enumerate}
    \item We formalize local-pass trace violations for tool-using agents,
    distinguishing action-local permission checks from global invariants over
    provenance, audience, tenant, memory, approval, and aggregation state.
    \item We introduce TraceBreak, a releaseable synthetic benchmark with 120
    paired risky/safe-control workflows across six compositional policy
    categories.
    \item We evaluate local guards, visible-content DLP, visible-policy and
    prompt-only baselines, same-action defense replays, and TraceGuard under
    deterministic and API-driven traces.
    \item We identify an important implementation boundary: provenance fields
    written by the model are convenient for benchmarking but are not a trustworthy
    enforcement substrate. Runtime-inferred provenance is the stronger design.
\end{enumerate}

\section{Problem Setting}

We consider an agent system that interacts with a synthetic enterprise
environment through structured tools. The world contains documents, people,
customers, support records, approvals, and shared memory. Tools include search,
reads, summarization, memory writes and reads, approval lookup, record
aggregation, email, ticket updates, and final answers. Email and ticket updates
are simulator sinks; no real external service is used.

A trace is a sequence
\[
\tau = [(a_1, t_1, x_1, y_1, m_1), \ldots, (a_T, t_T, x_T, y_T, m_T)],
\]
where $a_t$ is the actor, $t_t$ the tool, $x_t$ the tool input, $y_t$ the tool
output, and $m_t$ hidden runtime metadata such as provenance tags. A local guard
evaluates each proposed action with a function
$L(x_t, s_t) \in \{\mathrm{allow}, \mathrm{deny}\}$ over visible call-site
state $s_t$: the tool exists, object identifiers are valid, and referenced
observations exist. A global policy evaluates the whole trace with
$G(\tau,w) \in \{\mathrm{safe}, \mathrm{violation}\}$ over world state $w$ and
the executed events. A local-pass violation occurs when
\[
\forall t,\; L(x_t, s_t)=\mathrm{allow}
\quad\text{but}\quad
G(\tau,w)=\mathrm{violation}.
\]

TraceBreak does not rely on a malicious user prompt. Risky tasks are
benign-looking workflows whose requested sink is invalid under hidden enterprise
policy. We exclude trivial cases where a single action violates an ACL or where a
visible sink body contains an obvious literal secret. We also avoid LLM judges
for ground truth: a violation is counted only when the executed trace performs a
forbidden write, cross-tenant update, invalid approval use, or non-aggregate
export. This makes the benchmark narrow, but its labels deterministic.

\section{TraceBreak Benchmark}

TraceBreak is a pure-Python simulator with deterministic task generation and
matched counterfactual controls. It contains 120 tasks: 10 seeds for each of six
policy categories, with one risky task and one safe-control task per seed. The
safe control changes a single policy-relevant field---for example the recipient,
tenant, approval scope, or output form---while preserving the surface workflow.

The key benchmark design choice is that policy metadata is separated from visible
text. A document may visibly contain only renewal planning notes, while its hidden
tag records that it is confidential, belongs to AtlasCo, is intended for sales
and finance, and is not externally shareable. This separation tests whether a
defense tracks provenance and policy state, rather than whether it notices a
literal marker such as ``confidential.''

We evaluate two execution modes. Scripted traces replay canonical plans under
single-agent and multi-agent scaffolds. API traces use a language model that
selects one JSON tool call at a time from the user instruction and visible
observations. In the cooperative API setting, the model is instructed to include
\texttt{source\_refs} when writing transformed information to email, tickets,
memory, summaries, or aggregates. Later stress tests delete or corrupt these
references to separate the clean benchmark result from the stronger production
claim.

\section{TraceGuard and Baselines}

\paragraph{Local guards.}
The local guard validates the current action only: the tool is known, object IDs
exist, recipients and approvals exist, and declared source references are well
formed. It does not track where information came from or whether a later sink is
compatible with earlier sources.

\paragraph{Visible-content DLP.}
The DLP baseline blocks visible sensitive markers such as ``confidential'' or
``restricted'' and exact non-shareable document titles in external writes. It is
a deliberately simple baseline for text filtering without hidden provenance.

\paragraph{Visible policy and policy prompting.}
The deterministic visible-policy guard checks public sink-visible fields, such as
whether an approval visibly matches the requested recipient or whether an
external email references many outputs. The API policy-prompt condition instead
adds natural-language instructions forbidding unauthorized sharing, tenant
mixing, invalid approval reuse, and memory laundering. Neither receives hidden
source tags.

\paragraph{TraceGuard.}
TraceGuard attaches a \texttt{DataTag} to each tool output. Tags include source
IDs, derived-from IDs, sensitivity, tenant set, allowed audience, an
external-share flag, record count, and aggregate-only status. Transformations
such as summaries, memory writes, and memory reads merge tags conservatively:
sensitivity takes the maximum level, tenants and sources are unioned, audience
restrictions are intersected when present, and external sharing is allowed only
if all sources allow it. Sinks such as email and ticket updates check the merged
tag against recipient, tenant, approval, and aggregation invariants.

For email, TraceGuard blocks if non-shareable data reaches an external recipient
without a matching approval, if the recipient role is outside the allowed
audience, if tenant-specific data reaches another tenant, if an approval does not
match the object, recipient, purpose, or time, or if a detailed record list above
the aggregation threshold is exported externally. For ticket updates, it blocks
cross-tenant writes.

\paragraph{Runtime-inferred provenance.}
The clean API benchmark uses model-supplied \texttt{source\_refs}. This is useful
for exposing how trace monitors work, but not sufficient as a security boundary.
We therefore add offline replays that infer sink provenance from trusted runtime
events rather than trusting the sink's own references. This variant is the design
we would prefer in deployment: the runtime, not the model, owns the dependency
graph.

\section{Experiments and Results}

\paragraph{Protocol and metrics.}
We run a 120-task deterministic validation matrix and a 24-task API subset using
\texttt{gpt-4.1-mini}. API runs use cached responses, structured JSON actions,
temperature zero, a maximum of eight tool steps, and deterministic grading from
executed sink events. The main metrics are safe utility, the fraction of safe
controls completed without violation; risky violation, the fraction of risky
tasks where a forbidden sink executes; and local-pass violation rate (LPVR), the
fraction of risky tasks where all local checks pass but the trace violates
global policy. We keep detailed sink rates, block rates, category tables, and
token/cost fields in the artifact rather than overloading the main text.

\paragraph{Finding 1: local success is not trace safety.}
Local guards and content DLP complete all safe controls and all risky tasks in
both the deterministic suite and the API subset. That looks like high task
success, but every risky completion is a local-pass violation. The decisive fact
is usually not in the current action string: it is in the source tag, recipient
metadata, approval record, memory history, or aggregation state. This is the core
local-permission/global-policy gap.

\paragraph{Finding 2: prompting helps, but mostly by behavior change.}
The policy prompt reduces API risky violations from 100\% to 75\%, but safe
utility also drops from 100\% to 83\%. The policy-prompt diagnostic shows that
some avoided risky violations are abstentions that also lose matched safe
controls, while another changes the sink target rather than enforcing the
intended policy. Prompting is useful guidance, but it is not a reliable execution
boundary.

\paragraph{Finding 3: structural policy state is the missing object.}
TraceGuard blocks every risky sink attempt in the 120-task validation and the
24-task API subset while preserving all safe-control completions. The same-action
replay sharpens the conclusion: using the exact actions proposed by the API-local
agent, DLP still misses all risky traces, visible-policy review blocks only the
sink-visible approval and bulk-export subset, and both TraceGuard and a
metadata-aware critic block all risky sinks with no safe false blocks
(Table~\ref{tab:replay-summary}). The difference is not more natural-language
review; it is access to the policy state that local defenses lack.

\paragraph{Finding 4: provenance must be runtime-owned.}
A source-reference audit finds that all 92 executed write sinks in the cached
cooperative API traces carry nonempty, valid prior references. However, this
cooperation should not be trusted. When we replay TraceGuard after deleting sink
references, ordinary TraceGuard permits all risky replays; when we corrupt sink
references to benign public observations, it still permits most risky replays.
A conservative StrictTraceGuard restores safety for missing references but
heavily overblocks safe controls. Runtime-inferred provenance recovers the right
behavior on sink deletion and corruption: 100\% safe utility, 0\% risky
violations, and 100\% risky blocks. Erasing intermediate transform references
creates one laundering violation for ordinary TraceGuard; runtime inference
removes that violation with one safe false block. The production lesson is clear:
do not trust model-authored provenance fields when the runtime can infer or
validate dependencies itself.

\paragraph{Robustness checks.}
Supplementary audits support the same story without additional API spending.
Category tables show the API pattern across all six categories. The
block-reason audit finds category-aligned TraceGuard reasons and no safe-control
blocks. Deterministic authority-transfer, indirect-injection overlay, and
decoy-clutter stress tests preserve the structural gap: local and DLP defenses
miss risky local-pass violations, visible-policy defenses catch only visible
subsets, and TraceGuard or runtime-inferred TraceGuard blocks risky sinks while
preserving matched safe controls.

\paragraph{Example trace.}
In a sensitive-external task, the agent searches for AtlasCo renewal notes, reads
\texttt{doc\_pricing\_100}, summarizes it, finds an external recipient, and
sends an email. Local checks allow every step. The hidden tag on the email body
still contains \texttt{source=doc\_pricing\_100},
\texttt{sensitivity=confidential}, \texttt{tenant=AtlasCo}, and
\texttt{external\_share=false}. TraceGuard blocks the same final email with
reason \texttt{internal\_data\_to\_external\_recipient}. The artifact includes a
compact trace pair and per-category examples.

\paragraph{Reproducibility.}
TraceBreak uses synthetic data and deterministic labels. Tasks are generated from
seeds, traces are stored as JSONL, API responses are cached, and all headline
metrics are recomputed from executed sink events rather than LLM judges. The
artifact includes scripts for the deterministic suite, API subset, same-action
replays, provenance stress tests, category examples, claim-boundary audits, and
paper rebuilds.

\section{Related Work}

\paragraph{Tool-using agents and agent benchmarks.}
Early agentic prompting and tool-use work showed that language models can
interleave reasoning with actions and call external APIs \citep{yao2022react},
learn to use tools \citep{schick2023toolformer}, and interact with large API
collections \citep{patil2023gorilla}. This increased capability motivates
benchmarks for tool-use reliability and rule following, such as $\tau$-bench for
tool-agent-user interaction in realistic domains \citep{yao2024taubench}.
TraceBreak follows this line in treating tools and executed state as the unit of
evaluation, but focuses on trace-level security rather than task success alone.

\paragraph{Prompt injection and agent security benchmarks.}
Indirect prompt injection showed that LLM-integrated applications can confuse
untrusted external content with trusted instructions
\citep{greshake2023notwhat,liu2023promptinjection}. Follow-up work formalizes
prompt-injection attacks and defenses \citep{liu2023formalizingprompt}, and
OWASP identifies prompt injection, excessive agency, and sensitive information
disclosure as central LLM-application risks \citep{owasp2025llm01}. AgentDojo,
InjecAgent, ASB, and AgentSecBench benchmark attacks and defenses in tool-using
agents \citep{debenedetti2024agentdojo,zhan2024injecagent,zhang2024asb,alpay2026agentsecbench}.
Recent contextual-integrity analysis argues that future prompt-injection
failures can arise from manipulated context and inappropriate information flows,
not only from obvious imperative strings \citep{abdelnabi2026promptintegrity}.
TraceBreak is complementary: its risky tasks need not contain injected
instructions at all. They are policy violations caused by locally valid
composition.

\paragraph{Runtime enforcement and information flow.}
Our monitor is inspired by classic security ideas: least privilege and complete
mediation \citep{saltzer1975protection}, information-flow labels
\citep{denning1976lattice}, enforceable trace properties
\citep{schneider2000enforceable}, and confused-deputy failures
\citep{hardy1988confused}. Recent agent defenses bring related ideas into LLM
systems. Task Shield checks action alignment with the user task
\citep{jia2024taskshield}; MELON detects indirect prompt injection through
masked re-execution and tool comparison \citep{zhu2025melon}; RTBAS and Fides
adapt information-flow control to tool-based agents \citep{zhong2025rtbas,costa2025fides};
AgentSpec and SafeAgent propose runtime protection architectures
\citep{wang2025agentspec,liu2026safeagent}; and ClawGuard enforces
user-confirmed constraints at tool-call boundaries \citep{zhao2026clawguard}.
TraceGuard is intentionally narrower than these systems. Its purpose is to make
one failure mode---local permission without global trace policy---easy to
reproduce, measure, and audit with paired controls.

\paragraph{Memory, multi-agent, and compositional risk.}
Long-lived memory and retrieval create persistence across turns and agents;
AgentPoison shows that poisoned memory or knowledge bases can compromise LLM
agents without model training \citep{chen2024agentpoison}. Multi-agent systems
add delegation, partial observability, and authority-transfer risks
\citep{schroederdewitt2025multiagentsecurity,hammond2025multiagentrisks}.
Closest to TraceBreak are Policy-Invisible Violations, which studies hidden
policy facts \citep{wu2026policyinvisible}, and Beyond Single-Agent Alignment,
which studies context-fragmented multi-agent violations
\citep{wu2026contextfragmented}. TraceBreak should not claim that hidden policy
state or trace enforcement is new. Its contribution is a compact, releaseable,
paired benchmark with deterministic sink-event labels, same-action defense
replays, and provenance deletion/corruption audits that make the trust boundary
explicit.

\section{Limitations and Ethics}

TraceBreak is synthetic. This enables safe release, deterministic labels, and
matched controls, but it cannot capture all enterprise policy complexity.
TraceGuard uses hand-coded invariants and conservative tag propagation; richer
semantic transformations may require policy-specific declassification rules and
finer-grained labels. The clean API result assumes cooperative \texttt{source\_refs};
our provenance stress tests show why production monitors should validate or
infer dependencies rather than trust model-authored fields. Runtime inference is
promising in replay, but a live deployed system would still need careful
engineering around implicit dependencies, redaction, declassification, and
recovery after a block.

The API subset is preliminary evidence, not a model leaderboard. The zero
observed TraceGuard violations cover the implemented TraceBreak invariants, not
arbitrary enterprise policies. We do not use real emails, tickets, browsing,
credentials, tenants, or users. All data, people, documents, approvals, and
messages are synthetic. The benchmark is intended for defensive evaluation of
trace-level enforcement.

\section{Conclusion}

Tool-level permission is not compositional. In TraceBreak, every individual
action can be locally allowed while the complete trace violates a global policy
over provenance, audience, tenant, memory, approval, or aggregation state. The
main result is not that a small monitor solves agent security. It is that
prompting and action-local checks leave a measurable gap whenever policy facts
are distributed across the trace. Trace-level monitoring, especially with
runtime-owned provenance, provides a simple and reproducible way to close that
gap on this benchmark.
